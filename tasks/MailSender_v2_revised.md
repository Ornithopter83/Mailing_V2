# MailSender_v2 개발 문서

## 1. 프로젝트 방향

`MailSender_v2`는 기존 콘솔형 `NaraEmailSender`를 작업용으로 복사한 뒤, WinForms 데스크톱 애플리케이션으로 개편한다.

기존 프로젝트는 사본이므로 수정할 수 있다. 다만 아직 SMTP 발송, HTML 본문, 이미지 CID, 첨부파일, 텍스트 보고서 등 참조할 구현이 남아 있으므로, 초안 구현이 완성되고 필요한 코드가 모두 v2 구조로 이전되기 전까지는 보존한다.

초안 구현이 완성된 뒤 참조할 내용이 모두 이전되면 기존 사본은 삭제해도 된다.

기존 프로젝트는 다음 기능의 참고 구현으로만 사용한다.

- SMTP 설정 및 발송 방식
- HTML 메일 본문 생성
- 이미지 CID 삽입 방식
- 첨부파일 처리
- 발송 결과 텍스트 보고서 생성

v2의 핵심은 단순 엑셀 즉시 발송이 아니라, `Excel 업로드 -> DB 반영 -> 발송 대상 조회 -> 선택 발송 또는 발송완료 처리 -> 이력 관리` 흐름이다.

작업을 시작할 때는 반드시 다음 문서를 먼저 확인한다.

- `AGENTS.md`
- `CurrentWork.md`
- 현재 작업에 해당하는 `tasks/*.md`

새 명령이나 작업 추가는 `tasks/00_TEMPLATE.md` 형식으로 작성한 task 문서를 기준으로 진행한다.

## 2. 1차 개발 목표

1차 개발은 실제 운영에 필요한 최소 기능을 갖춘 WinForms 프로그램을 목표로 한다. 2026-07-02 기준으로 핵심 기능은 `MailSender_v2`에 구현되어 있으며, 이후 작업은 task 문서 단위로 보완한다.

- Excel 메일링 리스트 업로드
- 이메일 정규화 및 유효성 검사
- DB 저장 및 기존 데이터 갱신
- 발송 현황 대시보드 표시
- 발송 대상 조건 조회
- 사용자가 체크박스로 발송 대상 선택
- 메일 작성 및 SMTP 발송
- 메일을 실제로 보내지 않고 발송완료 처리
- 발송 이력과 수동 발송완료 이력 관리
- 결과 텍스트 저장
- 결과 Excel 저장
- 로그 팝업 표시
- SMTP 발송 진행 상태 팝업 표시
- DB 조회 대상 발송 결과를 SMTP 사용자에게 보고 메일로 전송
- 보고 메일에 원본 본문 이미지와 첨부 파일 포함

## 2-1. 현재 구현 상태

현재 구현 기준은 다음 문서에서 확인한다.

- `CurrentWork.md`: 최신 작업 결과 요약
- `tasks/09_발송결과를SMTP_User에게보고.md`: 발송 결과 보고 메일
- `tasks/10_발송진행상태팝업.md`: 발송 진행 상태 팝업
- `MailSender_v2/Database/supabase_schema.sql`: Supabase 테이블, RLS, anon 권한

주요 구현 상태:

- `MailSender_v2`는 .NET Framework 4.8 WinForms `WinExe` 프로젝트이다.
- `MainForm.cs`는 Designer 기반이 아니라 코드 기반 UI이므로 프로젝트 파일에서 `SubType=Code`로 유지한다.
- 첫 번째 탭은 `1. 발송 현황`이며, Supabase 기준 요약과 발송 현황 초기화를 제공한다.
- 두 번째 탭은 `2. 발송 대상 / 메일 작성`이며, 대상 조회, 수동 완료, SMTP 발송, 미리보기, 첨부, 이미지 CID 발송을 담당한다.
- SMTP 발송 후 선택 대상 목록은 화면에서 초기화하며, DB 목록은 자동 재조회하지 않는다.
- Supabase 접근은 anon 또는 publishable key 기준이며, service role key는 앱에 넣지 않는다.

## 3. 신규 프로젝트 구조

권장 프로젝트명은 `MailSender_v2`이다.

```text
EMailService
├─ NaraEmailSender          기존 콘솔형 메일 발송 프로젝트
├─ NaraInfoCollectorBot     기존 입찰공고 수집 프로젝트
└─ MailSender_v2            신규 WinForms 프로젝트
```

생성 방식:

1. `NaraEmailSender` 폴더를 `MailSender_v2`로 복사한다.
2. 솔루션 파일, 프로젝트 파일, AssemblyName, RootNamespace를 `MailSender_v2` 기준으로 정리한다.
3. 기존 콘솔 `Program.cs`는 참조 대상으로 분리하거나 서비스 클래스로 이전한다.
4. WinForms `MainForm`을 앱 진입 화면으로 만든다.
5. SMTP, HTML 본문, CID 이미지, 첨부파일, 보고서 생성 코드를 v2 서비스 구조로 옮긴다.
6. 참조 이전이 끝난 뒤 기존 사본 정리 여부를 판단한다.

권장 기술 구성은 다음과 같다.

- .NET Framework 4.8 또는 .NET 8 WinForms
- ClosedXML: Excel 읽기 및 결과 Excel 저장
- Newtonsoft.Json 또는 System.Text.Json: 설정 파일 처리
- Supabase: 원격 DB 및 발송 이력 저장
- System.Net.Mail: 기존 SMTP 발송 방식 유지

기존 프로젝트가 .NET Framework 4.8 기반이므로, 빠른 이식성을 우선하면 `MailSender_v2`도 .NET Framework 4.8 WinForms로 시작한다. 장기 유지보수를 우선하면 .NET 8 WinForms도 검토할 수 있다.

## 4. 전체 화면 구조

프로그램은 상단 탭 2개로 구성한다.

```text
입찰공고 메일링 시스템
├─ 1. 발송 현황
└─ 2. 발송 대상 / 메일 작성
```

역할은 명확히 분리한다.

- `발송 현황`: Supabase 기준 발송 현황 요약을 확인하고, 필요 시 발송 이력만 초기화한다.
- `발송 대상 / 메일 작성`: DB에서 발송 가능한 대상을 조회하고, 실제 발송 또는 수동 발송완료 처리를 수행한다.

발송 대상 조회와 SMTP 발송 진행 상황은 2번 탭과 발송 진행 상태 팝업에서 처리한다. 통계 차트, 예약 발송, 권한 관리는 1차 개발에서 제외한다.

## 5. Supabase DB 설계 초안

Supabase 테이블은 Supabase SQL Editor 또는 별도 마이그레이션 SQL로 관리한다.

데스크톱 앱에는 Supabase `service_role` key를 포함하지 않는다. 앱은 anon key와 RLS 정책을 전제로 접근하며, 민감한 키는 코드, 문서, 로그에 기록하지 않는다.

### 5-1. Recipients

메일 수신 대상 기본 정보 테이블이다.

|컬럼|설명|
|-|-|
|Id|PK|
|Email|원본 또는 표시용 이메일|
|NormalizedEmail|정규화 이메일, 중복 판단 기준|
|AgencyName|기관명|
|NoticeDate|공고일자|
|NoticeName|공고명|
|ManagerName|담당자|
|Phone|연락처|
|LastUploadedAt|마지막 업로드 일시|
|CreatedAt|최초 등록 일시|
|UpdatedAt|수정 일시|

`NormalizedEmail`에는 unique index를 둔다.

### 5-2. UploadHistory

엑셀 업로드 작업 이력 테이블이다.

|컬럼|설명|
|-|-|
|Id|PK|
|FileName|업로드 파일명|
|FilePath|업로드 파일 경로|
|SheetName|처리된 시트명|
|TotalRows|전체 데이터 행 수|
|InsertedCount|신규 등록 건수|
|UpdatedCount|기존 갱신 건수|
|EmptyEmailCount|이메일 없음 건수|
|InvalidEmailCount|이메일 형식 오류 건수|
|DuplicateCount|파일 내 중복 제거 건수|
|BlockedCount|차단 목록 제외 건수|
|ErrorCount|기타 오류 건수|
|CreatedAt|업로드 일시|

### 5-3. SendHistory

실제 SMTP 발송 및 수동 발송완료 처리를 모두 기록한다.

|컬럼|설명|
|-|-|
|Id|PK|
|RecipientId|수신 대상 ID|
|Email|처리 당시 이메일|
|Subject|메일 제목|
|Status|Sent, Failed, ManuallyMarkedSent|
|Method|Smtp, Manual|
|Memo|처리 사유 또는 실패 메시지|
|ProcessedAt|처리 일시|

`Method = Manual`은 메일을 보내지 않고 발송완료 처리한 경우이다.

### 5-4. BlockedEmails

차단 이메일 테이블이다.

|컬럼|설명|
|-|-|
|Id|PK|
|NormalizedEmail|정규화 이메일|
|Reason|차단 사유|
|CreatedAt|등록 일시|

## 6. 목록 추가 탭

### 6-1. 화면 목적

이 화면에서는 발송 대상 조회나 메일 작성 기능을 보여주지 않는다. 파일 업로드, DB 반영, 처리 결과 확인, 결과 저장, DB 초기화, 로그 확인 기능만 제공한다.

### 6-3. 엑셀 파일 업로드

구성 요소:

- 파일 경로 TextBox

동작:

## 7. 발송 대상 / 메일 작성 탭

### 7-1. 화면 목적

DB에 저장된 이메일 중 발송 가능한 대상을 조회하고, 메일을 작성하여 발송한다.

추가로 기존 방식으로 이미 발송한 대상이 있는 경우를 위해, 실제 메일을 보내지 않고 선택 대상을 발송완료 처리하는 기능을 제공한다.

### 7-2. 레이아웃

좌우 분할 구조를 사용한다.

```text
좌측 60%: 메일 작성
우측 40%: 발송 대상 조회
```

### 7-3. 메일 작성 영역

구성 요소:

- 제목 TextBox
- 참조 TextBox
- 본문 작성 영역
- 이미지 삽입 Button
- 첨부파일 목록
- 파일 추가 Button
- 삭제 Button
- 임시저장 Button
- 미리보기 Button
- 발송하기 Button

본문은 이미지 삽입이 필요하므로 단순 TextBox보다 RichTextBox 또는 HTML Editor 기반 구성을 검토한다. 1차 개발에서는 RichTextBox와 HTML 변환 방식으로 시작할 수 있다.

발송 전 검사:

- 선택된 발송 대상 수
- 제목 입력 여부
- 참조 이메일 형식
- 첨부파일 존재 여부
- 첨부파일 경로 유효성
- SMTP 설정 여부

### 7-4. 발송 대상 조회 영역

조회 조건:

- 공고일자 정렬
  - 내림차순
  - 오름차순
- 발송 건수
- 이미 발송된 이메일 제외
- 차단 목록 제외

기본값:

```text
공고일자 정렬: 내림차순
발송 건수: 50
이미 발송된 이메일 제외: 체크
차단 목록 제외: 체크
```

대상 목록 DataGridView 권장 컬럼:

|컬럼|설명|
|-|-|
|선택|체크박스|
|이메일|수신 이메일|
|기관명|공고기관 또는 수요기관|
|공고일자|공고일자|

필요 시 공고명, 담당자, 연락처를 추가할 수 있으나, 우측 영역 폭을 고려해 1차 개발에서는 컬럼을 과도하게 늘리지 않는다.

### 7-5. 발송하기

발송 버튼 클릭 시 별도 화면은 만들지 않고 MessageBox로 확인한다.

```text
발송 대상: 50명
제목: 입찰공고 안내드립니다

발송하시겠습니까?
```

`예`를 누르면 SMTP 발송을 시작한다.

발송 처리:

1. 발송 시작 로그 기록
2. 개별 수신자에게 SMTP 발송
3. 성공 시 `SendHistory.Status = Sent`, `Method = Smtp` 기록
4. 실패 시 `SendHistory.Status = Failed`, 실패 메시지 기록
5. 기존 방식의 텍스트 보고서 생성
6. 발신자용 결과 메일 발송은 기존 구현을 참고하되 설정으로 켜고 끌 수 있게 한다.

### 7-6. 발송완료 처리

신규 추가 기능이다.

이 기능은 실제 메일을 보내지 않고, 선택된 발송 대상에 대해 발송 이력만 등록한다. 기존 콘솔 방식이나 다른 수단으로 이미 발송한 이메일을 v2 DB에 반영하기 위해 사용한다.

권장 버튼명:

- `발송완료 처리`

권장 위치:

- 발송 대상 조회 영역 하단
- 또는 `발송하기` 버튼 근처에 보조 작업 버튼으로 배치

동작:

1. 사용자가 발송 대상 조회를 수행한다.
2. 처리할 대상을 체크박스로 선택한다.
3. `발송완료 처리` 버튼을 클릭한다.
4. MessageBox로 확인한다.
5. 선택된 대상에 대해 `SendHistory`에 수동 처리 이력을 등록한다.
6. 조회 조건에서 `이미 발송된 이메일 제외`가 체크되어 있으면 이후 조회에서 제외된다.
7. 작업 로그와 로그 팝업에 처리 결과를 표시한다.

확인 MessageBox 예시:

```text
선택된 대상 50건을 실제 메일 발송 없이 발송완료 처리합니다.

기존 방식으로 이미 발송한 대상에만 사용해야 합니다.
계속하시겠습니까?
```

처리 기록:

```text
Status = ManuallyMarkedSent
Method = Manual
Memo = 사용자가 발송 없이 발송완료 처리
ProcessedAt = 현재 시각
```

유효성 검사:

- 선택된 대상이 없으면 처리 불가
- 이미 발송 이력이 있는 대상은 중복 처리 여부를 확인
- 차단 목록 대상은 기본적으로 처리 대상에서 제외
- 처리 전 사용자의 최종 확인 필요

권장 로그:

```text
[10:32:10] 수동 발송완료 처리 시작: 50건
[10:32:11] 1/50 발송완료 처리: abc@korea.kr
[10:32:12] 2/50 발송완료 처리: def@korea.kr
[10:32:20] 수동 발송완료 처리 완료: 50건
```

## 8. 로그 팝업

로그 팝업은 업로드, 실제 발송, 수동 발송완료 처리에서 공통으로 사용한다.

구성 요소:

- 창 제목: 로그
- 진행 상태 문구
- 진행률 바
- 로그 TextBox 또는 ListBox
- 닫기 버튼

발송완료 처리 로그도 실제 발송 로그와 구분되도록 `수동 발송완료` 문구를 포함한다.

## 10. 이메일 정규화 및 제외 기준

이메일 비교는 원본 문자열이 아니라 정규화 이메일 기준으로 수행한다.

정규화 기준:

- 앞뒤 공백 제거
- 소문자 변환
- 빈 값 제외
- 이메일 형식 검사

중복 및 제외 기준:

1. 파일 내부 중복
2. DB 기존 이메일
3. 현재 발송 이력
4. 수동 발송완료 이력
5. 차단 목록

발송 제외 항목:

- 이메일 없음
- 이메일 형식 오류
- 파일 내 중복
- DB 차단 목록 포함
- 이미 SMTP 발송된 이메일
- 수동 발송완료 처리된 이메일

## 14. 최종 요약

`MailSender_v2`는 기존 `NaraEmailSender` 사본을 작업용으로 복사한 뒤 WinForms 프로젝트로 개편한다.

기존 콘솔 프로젝트는 발송 로직 참고용으로 유지하고, v2에서는 DB 기반으로 업로드, 조회, 발송, 이력 관리를 분리한다. 초안 완성 후 참조 이전이 끝나면 기존 사본은 삭제 가능하다.

특히 운영상 이미 발송된 대상이 존재하므로, 실제 메일을 보내지 않고 선택 대상만 `발송완료 처리`하는 기능을 1차 개발 범위에 포함한다. 이 기능은 `SendHistory`에 `Method = Manual`, `Status = ManuallyMarkedSent`로 기록하며, 이후 발송 대상 조회에서 이미 발송된 이메일 제외 조건에 포함된다.

앞으로 작업 추가는 `tasks/00_TEMPLATE.md` 형식의 번호형 task 문서를 기준으로 진행한다.
