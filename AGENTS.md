# EMailService Codex 작업 규칙

이 문서는 Codex가 `EMailService` 저장소에서 작업할 때 먼저 확인해야 하는 규칙이다.

## 기본 원칙

- 작업 전 관련 문서와 기존 프로젝트 구조를 먼저 확인한다.
- 큰 변경이나 여러 파일에 걸친 변경 전에는 짧은 계획을 먼저 제시한다.
- 사용자 요청이 문서 정리인 경우 실제 구현은 하지 않고 문서 범위만 수정한다.
- 기존 콘솔 프로젝트는 사본이므로 수정할 수 있다. 다만 참조할 구현을 먼저 확인하고 필요한 부분은 새 구조로 옮긴 뒤 정리한다.
- `MailSender_v2`는 기존 `NaraEmailSender`를 작업용으로 복사해 시작할 수 있다. 초안 구현이 완성되고 참조할 내용이 모두 이전되면 원본 사본은 삭제해도 된다.
- 신규 기능은 가능하면 `MailSender_v2` 프로젝트 안에서 구현한다.
- 작업 후 가능한 경우 빌드 또는 실행 확인을 수행하고, 수행하지 못한 경우 이유를 남긴다.
- 완료 보고에는 변경 파일과 변경 이유를 요약한다.

## 변경 제한

- 기존 `NaraEmailSender`의 SMTP 동작은 참조 후 이전한다. 이전이 끝나기 전에는 동작 의미를 임의로 바꾸지 않는다.
- 기존 `NaraInfoCollectorBot`의 입찰공고 수집 동작은 이번 메일러 작업 범위 밖이므로 임의로 변경하지 않는다.
- 기존 설정 파일 형식을 임의로 변경하지 않는다.
- DB 스키마 변경은 문서화된 설계 또는 사용자 승인 후 진행한다.
- 빌드 설정, 패키지, 배포 설정은 필요한 범위에서만 변경한다.
- WinForms `Designer.cs` 직접 수정은 최소화한다. 필요한 경우 레이아웃 영향 범위를 확인한다.
- UI 문구는 한국어 운영 환경을 우선하되, 추후 현지화 가능성을 해치지 않도록 정리한다.

## 보안과 로그

- SMTP 계정, 비밀번호, 토큰을 코드나 문서에 기록하지 않는다.
- Supabase service role key는 데스크톱 앱, 코드, 문서, 로그에 기록하지 않는다.
- Supabase 접근은 anon key와 RLS 정책을 전제로 한다.
- 실제 운영 이메일 목록은 로그와 문서에 필요 이상으로 남기지 않는다.
- 로그에는 발송 결과, 건수, 상태 중심으로 기록한다.
- 오류 로그에는 원인 파악에 필요한 메시지만 남기고 민감 정보는 제외한다.

## 리소스 관리

- Excel 파일, 첨부 파일, 이미지 파일, DB 연결, SMTP 연결은 사용 후 해제한다.
- WinForms UI 스레드와 백그라운드 작업 간 접근을 분리한다.
- 대량 발송이나 대량 업로드 작업은 UI가 멈추지 않도록 비동기 또는 백그라운드 처리한다.
- 첨부파일과 본문 이미지의 파일 경로 유효성을 발송 전 확인한다.

## 작업 기준 문서

- 새 작업 전 `AGENTS.md`, `CurrentWork.md`, `MailSender_v2_revised.md`를 먼저 읽는다.
- 실제 작업은 `tasks/*.md` 파일 단위로 나누어 수행한다.
- 새 명령이나 작업 추가는 `tasks/00_TEMPLATE.md` 형식을 따른다.
- 작업 문서는 번호형 파일명(`00_...`, `01_...`)을 사용한다.
- 구현 작업을 시작하기 전에 해당 task 문서의 목표, 변경 금지, 요구사항, 완료 조건을 확인한다.
- 작업 완료 후 해당 task 문서의 `결과` 섹션을 최신 상태로 갱신한다.
- 새로 결정된 사항은 `CurrentWork.md`에 짧게 반영한다.

## 현재 주요 구현 방향

- `MailSender_v2`는 WinForms 데스크톱 프로젝트로 구성한다.
- 시작 방식은 기존 `NaraEmailSender`를 `MailSender_v2`로 복사한 뒤 개편하는 방식을 우선한다.
- 기존 `NaraEmailSender`는 SMTP 발송, HTML 본문, 이미지 CID, 첨부, 텍스트 보고서 참고용으로 사용하고, 참조가 끝난 코드는 v2 구조로 이전한다.
- v2의 핵심 흐름은 `Excel 업로드 -> Supabase DB 반영 -> 발송 대상 조회 -> 실제 발송 또는 수동 발송완료 처리 -> 이력 관리`이다.
- 발송완료 처리는 실제 메일을 보내지 않고 기존 발송 이력을 DB에 반영하기 위한 1차 핵심 기능으로 포함한다.
- 1차 개발에서는 웹페이지, 대시보드, 예약 발송, 통계 차트, 권한 관리는 제외한다.
